The ELF (Executable and Linkable Format) file format is used for executables, shared libraries, object files, and core dumps on Linux.

Every ELF file is one of four types:

| Type | Description |
|---|---|
| `ET_REL` | Relocatable object (`.o`) — output of the compiler, not yet linked |
| `ET_EXEC` | Executable — statically linked, fixed load address |
| `ET_DYN` | Shared object (`.so`) / PIE — position-independent, loaded anywhere |
| `ET_CORE` | Core dump — snapshot of a crashed process |

The same format serves all four. What changes is which headers are present and how the OS interprets them.

You can inspect any ELF file with:

```bash
readelf -h ./binary      # ELF header
readelf -S ./binary      # section headers
readelf -l ./binary      # program headers (segments)
file ./binary            # quick type identification

```
We begin with the following code:

```c
// hello.c
#include <stdio.h>

int main() {
    printf("Hello, world!\n");
    return 0;
}
```

Compiled for x86_64 with:

```bash
docker run --rm --platform linux/amd64 -v $(pwd):/work -w /work gcc gcc hello.c -o hello
```

The first 16 bytes are the ELF Identification (`EI_*`). The magic bytes `\x7fELF` let the kernel identify ELF files without relying on file extension. The other fields encode endianness, ELF version, and OS ABI.

**`e_type`** — what kind of ELF file this is: `ET_REL` (`.o`, not yet linked), `ET_EXEC` (statically linked, fixed load addresses), `ET_DYN` (shared library or PIE — position-independent), or `ET_CORE` (core dump). Our `hello` binary shows **`ET_DYN`** because GCC produces PIE executables by default.

**`e_machine`** — the target architecture. `0x3e` = x86-64, `0x28` = ARM, `0xb7` = AArch64, `0xf3` = RISC-V. The kernel checks this against the running CPU before executing.

**`e_entry`** — virtual address the kernel jumps to after loading. For C programs this is **`_start`** (the CRT stub that calls `main`), not `main` itself.

**`e_phoff`** and **`e_shoff`** — byte offsets from the start of the file to the program header and section header tables. Either can be 0 if not present (e.g. `e_shoff = 0` in a stripped executable).

**`e_flags`** — architecture-specific flags. x86-64 leaves this **0** (nothing to encode). ARM uses it to record the ABI version and whether the code is Thumb-only.

**`e_ehsize`** — size of this header in bytes. Always **64** for 64-bit ELF, **52** for 32-bit.

**`e_phentsize`** and **`e_shentsize`** — size in bytes of a single entry in the program header or section header table. Fixed by the spec (**56** bytes for phdrs, **64** bytes for shdrs in 64-bit ELF).

**`e_phnum`** and **`e_shnum`** — count of entries in each table. Either can be 0: a `.o` file typically has 0 program headers; a stripped binary can have 0 section headers and still run. If `e_shnum` is 0 but `e_shoff` is non-zero, the actual count is in `sh_size` of the first section header.

**`e_shstrndx`** — index of the section header whose content is `.shstrtab`, the string table for all section names. Every section header stores its name as an offset into this table rather than inline.

Field sizes vary only by class (32-bit vs 64-bit), not by architecture — x86, ARM, RISC-V all follow the same layout within their class:

| Field | 32-bit | 64-bit |
|---|---|---|
| `e_entry` | 4 | **8** |
| `e_phoff` | 4 | **8** |
| `e_shoff` | 4 | **8** |
| **Total** | **52** | **64** |

Only `e_entry`, `e_phoff`, and `e_shoff` widen — they hold addresses and file offsets, so they grow with the address space.

### Pt 2: Section Headers and Sections VS Program Headers and Segments ###

The ELF binary contains a Section Header Table as well as a Program Header Table.

The Section Header Table's entries contain metadata regarding sections.  
They are used by the linker to identify where the code and data are, so it can stitch together multiple object files correctly.  

Sections are the linker's view of the file — they exist to organise code and data at link time. A stripped binary can have no section headers at all and still run.

The Program Header Table's entries are called segments, which let the kernel know at program load time, "what do I map into memory, where, and with what permissions (R/W/X)".

As such, there can be overlap in the chunks specified as segments and sections, as they serve different purposes.

Each section header (`Elf64_Shdr`) records:

| Field | Description |
|---|---|
| `sh_name` | Offset into `.shstrtab` — the section's name as a string |
| `sh_type` | `SHT_PROGBITS` / `SHT_SYMTAB` / `SHT_STRTAB` / `SHT_RELA` / … |
| `sh_flags` | `SHF_ALLOC` (loaded into memory) \| `SHF_EXECINSTR` \| `SHF_WRITE` |
| `sh_addr` | Virtual address (0 if not loaded into memory) |
| `sh_offset` | Byte offset in the file |
| `sh_size` | Size in bytes |
| `sh_link` / `sh_info` | Type-specific — for `.symtab`, `sh_link` = index of `.strtab` |
| `sh_addralign` | Alignment requirement |
| `sh_entsize` | Entry size for fixed-size tables (symtab, rela) |

`.bss` has `sh_type = SHT_NOBITS` — it takes up zero bytes in the file but `sh_size` bytes of memory at load time. The OS zero-fills it.

### Pt 3: Segments (Program Headers) ###

Segments are the OS loader's view of the file — they describe what to map into memory and how. The program header table exists only in `ET_EXEC` and `ET_DYN` ELF files.

One segment typically spans several sections. For example, a `LOAD` segment marked `R|X` covers `.text` and `.rodata`; a `LOAD` segment marked `R|W` covers `.data` and `.bss`.

<mark>Common segment types:</mark>

| Type | Description |
|---|---|
| `PT_LOAD` | Map this region into the process's virtual address space |
| `PT_DYNAMIC` | Points to the `.dynamic` section (dynamic linking info) |
| `PT_INTERP` | Path to the dynamic linker (`/lib64/ld-linux-x86-64.so.2`) |
| `PT_NOTE` | Build ID, ABI tag |
| `PT_GNU_STACK` | Controls whether the stack is executable (NX bit) |
| `PT_GNU_RELRO` | Region to remap read-only after relocation (hardening) |

<mark>Each program header entry (`Elf64_Phdr`) has the following fields:</mark>

| Field | Description |
|---|---|
| `p_type` | `PT_LOAD`, `PT_DYNAMIC`, etc. |
| `p_flags` | `PF_R` \| `PF_W` \| `PF_X` — read/write/execute permissions |
| `p_offset` | Byte offset in the file |
| `p_vaddr` | Target virtual address |
| `p_filesz` | Bytes to copy from the file |
| `p_memsz` | Bytes to allocate in memory (`p_memsz >= p_filesz`; excess is zero-filled — this is how `.bss` works at the segment level) |
| `p_align` | Alignment, typically `0x200000` (2 MB, matching the huge-page size) |

The kernel mmap's each `PT_LOAD` segment directly. For a PIE binary, the base address is randomised (ASLR) and all virtual addresses are rebased at load time.

The first entry in `hello`'s program header table is `PT_PHDR` — the entry that describes the program header table itself. Raw bytes:

### Pt 4: Sections (Section Headers) ###

Sections are the linker's view of the file. They carve the binary into named, typed chunks so that linkers and profilers know where code and read-only data is, where relocations need to be applied, etc.  
A fully stripped binary can have its section header table removed entirely and still run — the kernel only needs program headers.

Common sections:

| Section | Flags | Contents |
|---|---|---|
| `.text` | `R+X` | Executable machine code |
| `.rodata` | `R` | Read-only data — string literals, `const` arrays |
| `.data` | `R+W` | Initialised read-write globals (`int x = 5;`) |
| `.bss` | `R+W` | Uninitialised globals — zero-filled at load, **no bytes in file** |
| `.plt` | `R+X` | Procedure Linkage Table — stubs for lazy dynamic symbol resolution |
| `.got` | `R+W` | Global Offset Table — holds resolved addresses at runtime |
| `.dynamic` | `R+W` | Dynamic linking metadata (`DT_NEEDED`, `DT_SYMTAB`, …) |
| `.symtab` | — | Full symbol table — usually stripped from production binaries |
| `.dynsym` | — | Minimal symbol table needed at runtime for dynamic linking |
| `.strtab` | — | String table for `.symtab` symbol names |
| `.dynstr` | — | String table for `.dynsym` symbol names |
| `.shstrtab` | — | String table for section names themselves |
| `.rela.text` | — | Relocation entries for `.text` — addresses the linker must patch |
| `.debug_*` | — | DWARF debug info — present only in debug builds |

### Pt 5: The Symbol Table ###

The symbol table maps names to addresses. There are two:

- `.symtab` — full table, includes local symbols, usually stripped from release binaries
- `.dynsym` — minimal subset used at runtime, always present in shared objects

<mark>Each entry (`Elf64_Sym`, 24 bytes) has the following fields:</mark>

```
st_name   — offset into the associated string table
st_info   — high nibble: binding | low nibble: type
st_other  — visibility (STV_DEFAULT / STV_HIDDEN / STV_PROTECTED)
st_shndx  — section index the symbol lives in (SHN_UNDEF = imported)
st_value  — address (or offset in .o files)
st_size   — size in bytes
```

Binding:

```
STB_LOCAL   — not visible outside the object file (static functions/vars)
STB_GLOBAL  — visible to the linker across all objects
STB_WEAK    — global but can be overridden by a non-weak definition
```

Type:

```
STT_FUNC    — function / code
STT_OBJECT  — data object (variable, array)
STT_NOTYPE  — unspecified (labels, section starts)
STT_SECTION — represents a section itself
```

A symbol with `st_shndx = SHN_UNDEF` is an import — this object references it but doesn't define it. The linker must find the definition elsewhere.
