Measurements of Cache Effects: Part 2

12 Apr, 2026

This is a continuation from part 1, which condenses information from Ulrich Drepper’s paper - “What Every Programmer Should Know About Memory”. We cover the 3 chapter on CPUs.

The Test

We want to measure how access pattern (sequential vs random) and working set size interact with the cache hierarchy.

Sequential: each element’s n points to the next element in memory order, so pointer chasing happens to walk linearly through the array Random: the n pointers are shuffled, so each dereference jumps to an unpredictable location

By using a linked list traversal (always following pointer n), both tests do identical work – the only variable is whether the next element is at the predictable next memory address or a random one. This isolates exactly one thing: whether the prefetcher can anticipate your next memory access.

struct l {
    struct l *n;
    long int pad[NPAD];
};

By always advancing via the pointer rather than incrementing an index, both tests do identical work – the only variable is whether the next memory access is predictable. This isolates exactly one thing: whether the hardware prefetcher can anticipate your next access and hide latency.

The pad field controls element size. With NPAD=0, each struct is 8 bytes and 8 elements share a cache line. With NPAD=7, each struct is exactly one cache line, so every iteration requires a fresh fetch. This lets us measure the effect of spatial locality independently of working set size.

The working set is varied to push data through each cache level: L1d (16KB = 2^14 bytes), L2 (1MB = 2^20 bytes), and main memory beyond that, letting us observe miss costs and prefetcher effectiveness at each level.
